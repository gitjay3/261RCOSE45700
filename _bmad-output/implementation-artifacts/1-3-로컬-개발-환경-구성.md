# Story 1.3: 로컬 개발 환경 구성

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

개발자로서,
AWS 없이 로컬에서 Redis와 PostgreSQL을 구동할 수 있는 개발 환경이 구성되기를 원한다,
그래서 외부 인프라 의존 없이 개발과 테스트를 진행할 수 있다.

## Acceptance Criteria

1. **Given** Docker Desktop이 설치된 로컬 머신에서, **When** `docker compose -f infra/docker-compose.yml up -d`를 실행하면, **Then** Redis 컨테이너가 포트 6379로 실행되고 `appendonly yes` 설정이 적용된다.
2. Redis DB0~3 역할(`REDIS_MQ_DB`, `REDIS_DEDUP_DB`, `REDIS_RATELIMIT_DB`, `REDIS_CACHE_DB`)이 `docker-compose.yml` 환경변수로 명시된다.
3. PostgreSQL 컨테이너가 포트 5432로 실행되고 `tracker` 데이터베이스가 생성된다.
4. `infra/.env.example`에 모든 필수 환경변수 목록(`VARCO_API_KEY`, `AWS_REGION`, `S3_BUCKET_NAME`, `CRAWL_INTERVAL_MINUTES`, `REDIS_*`, `DB_*`, `SERVICE_NAME`)이 주석과 함께 제공된다.
5. `.gitignore`에 `.env`, `*.env.local`, `__pycache__/`, `.gradle/`, `node_modules/`, `build/`, `dist/`가 포함된다.
6. `docker compose up -d` 후 Redis `PING` 응답이 `PONG`이고 PostgreSQL `\l` 명령으로 `tracker` 데이터베이스가 확인된다.

### docker-compose.yml 핵심 설계 결정

**Redis `appendonly yes` 필수 이유 (NFR13)**
Redis AOF(Append-Only File) 설정이 없으면 EC2 재시작 시 `posts:processing`에 있는 미완료 메시지가 유실된다. `command: redis-server --appendonly yes`로 컨테이너 기동 시 강제 적용.

**named volume 필수 이유**
`tmpfs` 또는 bind mount 없이 기본 컨테이너 레이어를 사용하면 `docker compose down` 시 데이터가 삭제된다. `redis_data`와 `postgres_data` named volume으로 `down/up` 주기에 데이터 보존.

**PostgreSQL `POSTGRES_DB=tracker` 명시**
미명시 시 기본 DB는 `POSTGRES_USER`와 동일한 이름으로 생성됨. Story 1.4 Flyway 마이그레이션이 `tracker` DB를 타겟으로 하므로 명시 필수.

**x-redis-db-config YAML 앵커 패턴**
```yaml
x-redis-db-config: &redis-db-config
  REDIS_MQ_DB: ${REDIS_MQ_DB:-0}
  REDIS_DEDUP_DB: ${REDIS_DEDUP_DB:-1}
  REDIS_RATELIMIT_DB: ${REDIS_RATELIMIT_DB:-2}
  REDIS_CACHE_DB: ${REDIS_CACHE_DB:-3}
```
이후 crawler/detection/api 서비스 추가 시 `<<: *redis-db-config`으로 재사용. Epic AC "docker-compose.yml 환경변수로 명시" 요건 충족.

### `shared/config/redis_config.py`와의 관계

Story 1.2에서 구현된 `shared/config/redis_config.py`에 `REDIS_MQ_DB=0` 등 상수가 이미 정의되어 있다.
docker-compose.yml의 `REDIS_MQ_DB` env var 값과 `redis_config.py`의 상수가 **반드시 일치**해야 한다.
- `shared/config/redis_config.py`: `REDIS_MQ_DB: int = 0` (하드코딩 상수 — 앱 레이어 기본값)
- `infra/.env.example`: `REDIS_MQ_DB=0` (환경변수 — 운영 환경 오버라이드 가능)
- Story 1.2 Review Deferred: "Redis key 상수 네임스페이스 없음" → 이번 스토리에서 env var 정의로 일부 완화

### Anti-Patterns to Avoid

1. ❌ **`docker-compose.yml`에 실제 비밀값 하드코딩** — `POSTGRES_PASSWORD=supersecret`처럼 직접 작성 금지. `${DB_PASSWORD}` 환경변수 참조 사용.
2. ❌ **Redis `--appendonly no` (기본값)** — 재시작 시 posts:processing 메시지 유실. `appendonly yes` 필수.
3. ❌ **`infra/.env` 생성 후 git 커밋** — `.env.example`만 커밋. `.env`는 `.gitignore`에 의해 제외됨.
4. ❌ **Flyway 마이그레이션 SQL 미리 작성** — Story 1.4 범위. `db/migration/` 파일 생성 금지.
5. ❌ **`infra/prometheus/prometheus.yml` 작성** — Story 5.1 범위. 현재 `.gitkeep` 유지.
6. ❌ **Redis DB0~3을 별도 Redis 컨테이너로 분리** — 논리 분리(DB number)이며 물리 분리 아님. 단일 Redis 컨테이너.
7. ❌ **docker-compose v1 문법(`version: '3.8'` 헤더)** — Docker Compose v2에서 `version` 키 deprecated. 최상위 `services:` 바로 시작.

### 검증 명령 참조

```bash
# 컨테이너 기동
docker compose -f infra/docker-compose.yml up -d

# 컨테이너 상태 확인
docker compose -f infra/docker-compose.yml ps

# Redis 검증
docker compose -f infra/docker-compose.yml exec redis redis-cli PING
# 예상: PONG
docker compose -f infra/docker-compose.yml exec redis redis-cli CONFIG GET appendonly
# 예상: appendonly / yes

# PostgreSQL 검증
docker compose -f infra/docker-compose.yml exec postgres psql -U tracker_user -d tracker -c "\l"
# 예상: tracker 데이터베이스 목록에 표시

# 정리
docker compose -f infra/docker-compose.yml down
# 볼륨 포함 완전 정리 (주의: 데이터 삭제)
docker compose -f infra/docker-compose.yml down -v
```
